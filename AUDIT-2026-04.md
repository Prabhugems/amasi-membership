### 2.2 Detected patterns and structural fixes

#### Pattern 1 — Middleware allowlist instead of opt-in handler auth
**Symptom:** Every new public route ships 401-blocked and is later "fixed" by adding a string. `dcf98a8` (`/api/credential`), `0c68e75` (resume-from-token, `/api/r/`), `737ec5b` (`/api/applications/save-draft` was **silently 401 for 19 days** since `7376ffd` — server-side draft persistence was broken because the 401 envelope mimicked the route's own error envelope), `b788a3d` (Zoho OAuth), `e162e49` (upgrade module), `e2cc5f2` (`/members` matched `/member` because of `startsWith`).

**Root cause:** Middleware uses opt-out (admin cookie → all `/api/*` unless listed). The allowlist is in `src/middleware.ts`; handlers live elsewhere. No compile-time linkage.

**Structural fix:** Invert. Routes opt **in** to auth via wrappers (`withAdminAuth(handler)`, `withMemberAuth(handler)`). Middleware does only rate-limit + CSRF + logging. Add a Playwright test that hits every `src/app/api/**/route.ts` unauthenticated and asserts the response code matches the route's declared auth.

#### Pattern 2 — `apply/page.tsx` is a 2,985-line state monolith with silent fallbacks
**Symptom:** `c86d724` review fields disappear after one keystroke; `0b21cd3` follow-up violated Rules of Hooks; `8855294` 95% of draft applicants trapped at step-2 OTP loop because `setPhase(selectedType ? "upload" : "landing")` silently bounced null-type users; `467c050` zombie null-type drafts; `da2faab`/`60275a1`/`95df9b6` payment-vs-draft race conditions; `54cc24e` 4-phase payment-without-submission fix.

**Root cause:** 24 fix commits on one file is the smoking gun. Phase orchestration, OTP, OCR, draft sync, payment, and review share `useState` closures. The "22 zombie draft rows over 3 weeks" memory and the AGENTS.md "crash loudly, not redirect silently" rule both came from this exact file. Each fix adds another guard; the underlying state graph is unchanged.

**Structural fix:** Extract phase orchestration into `useReducer` (or XState) with **explicit invariants** per phase: a transition function that throws when `selectedType == null && nextPhase === "upload"`. Co-locate validation with transitions, not render. Replace `useCallback` closures over async state with refs or machine context.

#### Pattern 3 — Approval/submit forgets fields when copying application → member
**Symptom:** `cc8012c` document URLs (MCI cert, PG, ASI) never copied — members showed all docs as "Missing"; `4e8679e` profile_photo never copied; `5169bc3` profile photos never even uploaded post-face-detection (only stored as `File` in React state); `93b87a4` approve route 500 — missing import; `28bfee6` "21 bugs from end-to-end data flow trace" in one pass — bug #5 used `MAX(amasi)+1` instead of atomic sequence (race condition); bug #6 admin approve didn't roll back on update failure.

**Root cause:** Three handlers (`/api/applications/submit`, `/api/applications/approve`, `src/lib/auto-approval.ts`) each construct the member row independently. Adding a field to one doesn't propagate.

**Structural fix:** Single `buildMemberRowFromApplication(app): MemberInsert` pure function. All three approval paths import. Snapshot test asserts every column in `members` is either set by this function or explicitly excluded.

#### Pattern 4 — Schema drift kills writes silently because every insert is `try/catch + console.error`
**Symptom:** `7b76013` `membership_audit_log` renamed `target_type/target_id/details` → `entity_type/entity_id/new_data` — **11 callers silently failed**. Profile-update campaign sent 6,506 emails but stats sat at 0. `3227219`/`294284e` `members/search` referenced columns that don't exist; "Know Your Membership" returned nothing for every user. `8a6e8a7` `scoreApplication` reads camelCase form keys; resubmit passed snake_case DB rows → every score collapsed to ~5%.

**Root cause:** Two compounding habits: (1) `try { await supabase.insert(...) } catch (e) { console.error(e) }` everywhere — failed writes never surface as 500s. (2) Hand-typed column names with no compile-time check against schema.

**Structural fix:** `supabase gen types typescript` → commit as `src/types/database.types.ts` → `SupabaseClient<Database>` everywhere. Column-name typos become TS errors. Single `logAuditEvent()` helper. Audit-write failures emit to Sentry with table tag.

#### Pattern 5 — Razorpay Route + payment idempotency: implemented twice, wrong both times
**Symptom:** `e1465e3` Route transfer moved from `payments.transfer()` (returned "transfer not supported") to order-creation payload — **after** `af44d99` added "detailed Route transfer logging" to debug it. `60275a1` orphaned-payment detection. `da2faab` 3x submit retry. `f5af060` duplicate-payment guard + `payment_on_hold` widget. `54cc24e` 4-phase fix: `has_verified_payment` never set by save-draft → resume charged users twice.

**Root cause:** Payment treated as a side-effect of the form flow rather than a state machine with its own ledger. No single "payment intent" row atomically tracking `(order_id, payment_id, application_id, transferred_at, status)`.

**Structural fix:** `payment_intents` table keyed by `razorpay_order_id` with state column (`created → paid → applied → transferred → settled`). All routes read/write this row. Idempotency is "did we advance past state X for this order_id?" — one query. Orphan detection / duplicate guard / has_verified_payment collapse into the state column.

#### Pattern 6 — Next 16 client-router hooks without `<Suspense>`
**Symptom:** `7036f2f` `IncompletePage` uses `useSearchParams()` without Suspense → **all four prior commits silently failed to deploy for ~3 hours**, CDN served stale code. Introduced in `f5af060` adding `?status=` deep-linking. AGENTS.md was then updated with "always run `npx next build` locally."

**Root cause:** Static prerender failure surfaces only at `next build` on Vercel. The rule is prose, human-enforceable.

**Structural fix:** GitHub Action that runs `next build` on every PR and is required for merge. Or `.husky/pre-push` for changes under `src/app/**`. Mechanical, not aspirational.

#### Pattern 7 — Tickets module: auth bolted on, then re-bolted (5+ "bypass" fixes in two days)
**Symptom:** 2026-04-18: `47cd2a0`, `562a983` remove email-param auth ("pre-existing vulnerability"); `bc86b9a` field-stripping; `165038c` strip admin-only fields (csat_token, assigned_to leaking); `574f52c` middleware regex `/api/tickets/[^/]+(/reply)?` accidentally allowed `/api/tickets/upload` and `/analytics` unauthenticated.

**Root cause:** Tickets shipped without role-aware response shapes — every handler does `select(*)` and the admin↔member difference is a manual field strip per route.

**Structural fix:** `AdminTicket` / `MemberTicket` types + `toMemberView(ticket): MemberTicket`. Leaking `csat_token` becomes a TS error. Combined with Pattern 1, the entire 2026-04-18 batch becomes impossible.

### 2.3 Bottom line

**Half of every commit shipped is a fix.** Three structural roots account for most: (1) middleware opt-out allowlist, (2) `apply/page.tsx` state monolith, (3) approval-side application→member field-copy duplication. Fixing any one would eliminate ~20-40 commits/month of recurring work. Fixing all three before AMASICON 2026 traffic is the highest-leverage move available.

## 4. Security audit

> **Three CRITICAL, five HIGH, six MEDIUM. Total remediation: ~5 engineer-days.**

### 4.1 CRITICAL

**C1. All four storage buckets are public-read** (`uploads`, `form-uploads`, `event-assets`, `downloads`).
Member medical certificates uploaded to `members/{uuid}/...` are fetchable directly via `https://jmdwxymbgxwdsmcwbahp.supabase.co/storage/v1/object/public/uploads/...` without auth. The careful IDOR check at `src/app/api/members/signed-url/route.ts:34-55` is bypassed because no signature is needed. Recently approved members' UUIDs can be inferred via the public `/api/credential` endpoint by crawling sequential `amasi_number`.
**Fix:** Set `uploads`/`form-uploads` to `public=false`, add service-role-only RLS on `storage.objects`, route all client reads through `/api/members/signed-url`.

**C2. RLS disabled on 15 sensitive tables** including `admin_users` (bcrypt hashes + TOTP secrets), `api_keys`, `otp_codes`, `support_tickets`, `ticket_replies`, `member_clinics`, `member_credentials`, `member_experiences`, `admin_audit_log`. All current routes use the service role (which bypasses RLS), so the gap is latent — but `src/hooks/use-realtime-count.ts:22` already uses the browser client. One edit away from accidental exfiltration.
**Fix:** One migration enabling RLS + service-role-only policies on all 15 tables. Highest first: `admin_users`, `api_keys`, `otp_codes`, `admin_audit_log`.

**C3. Self-issuing super-admin via env var bypasses 2FA.**
`src/app/api/auth/login/route.ts:42-77`. `ADMIN_DEFAULT_EMAIL=admin@amasi.org` + `ADMIN_DEFAULT_PASSWORD=Amasi@2026` in `.env.local`. This branch runs **before** the DB check, **bypasses TOTP entirely**, and grants `permissions: ["all"]`. Anyone reading `.env.local` (or its Vercel mirror) has a permanent superuser credential no DB rotation can revoke without a code change. The password is also trivially guessable.
**Fix:** Remove the env-admin path, or require TOTP for it. Rotate the password and treat current value as compromised.

| H1 | Cron endpoints likely 401'd by middleware (no `/api/cron/*` allowlist entry; routes verify Bearer but middleware blocks first). Verify Vercel cron logs. | `src/middleware.ts` |

**No `maxDuration` overrides anywhere.** Every route inherits Vercel Fluid Compute default (300s on Pro). A stuck Anthropic call burns 5 min of Fluid Compute and ties up function concurrency.

| Route | Risk |
|---|---|
| `cron/cleanup-drafts/route.ts` | High — sequential Razorpay + admin email blasts |
| `payments/check-existing/route.ts` | High — uncapped Razorpay fetches |
| `applications/submit/route.ts` | Medium-High — Claude scoring + email + WhatsApp |
| `api/ocr/route.ts` | Medium — Claude Vision, 15-20s tail for PDFs |

**Add `export const maxDuration = 60`** to user-facing routes; keep crons at 300s.

### 5.4 Razorpay webhook idempotency hole

`src/app/api/webhooks/razorpay/route.ts` line 60 checks idempotency on `payment.captured` (good), but the **auto-approval branch at line 168 is not idempotent against partial failure** — a retried webhook could double-create members on a flaky run. Razorpay retries on non-2xx.
